Relatum

Pure relational closure engine.

Relatum is a forward-chaining inference engine that operates exclusively on relations. There are no predefined mathematical semantics — equality, order, membership, and any other concept you need are expressed as plain relations with configurable properties.

Core Ideas

  • Relations first. Everything is a named relation between terms: equiv(a, b), lt(x, y), knows(alice, bob).
  • No built-in equality. Want equivalence? Declare it: rel equiv/2 : equivalence. This gives you symmetry, transitivity, reflexivity, and congruence.
  • Rules derive facts. Write inference rules like rule trans: lt(?x, ?y), lt(?y, ?z) |- lt(?x, ?z) and the engine will apply them until no new facts can be derived.
  • Membership. Any element can contain others (<Person> alice, bob). Relations can restrict which container each argument must belong to.
  • Declarative language. Programs are concise descriptions: declare elements, relations, state facts, and say derive.

What It Looks Like

# A minimal Relatum program

<element> a, b, c

rel equiv/2 : equivalence

equiv(a, b)
equiv(b, c)

derive

Output: the engine derives equiv(a, c), equiv(c, a), equiv(b, a), equiv(c, b), and all reflexive pairs equiv(a, a), equiv(b, b), equiv(c, c).

Quick Start

Write your first Relatum program in under a minute.

1. Open the App

Go to the web app. You'll see a code editor on the left and an output panel on the right.

2. Write a Program

Type (or load the "Equivalence" example):

<element> a, b, c

rel equiv/2 : equivalence

equiv(a, b)
equiv(b, c)

derive

3. Press Run

Click Run (or press Ctrl+Enter). The output panel shows:

  • Declarations and auto-generated rules
  • Initial facts you provided
  • Derived facts (highlighted in green)
  • Summary statistics: rounds, total facts, saturation status

4. Experiment

Try adding more constants, changing properties, or writing custom rules. The engine recomputes from scratch on each Run.

Tip: Use the examples dropdown in the editor to load ready-made programs demonstrating different features.

Elements

The atoms of your universe.

Everything in Relatum is an element. Elements are declared by placing them inside a container using angle-bracket syntax.

Syntax

# Declare elements in the root container
<element> a, b, c

# Declare elements inside a named container
<Person> alice, bob, carol
<City> paris, london

The Root: <element>

There is one built-in intrinsic element: element. Every declared element automatically belongs to element. Writing <element> a, b, c is the simplest way to introduce elements into the universe.

Any Element Can Be a Container

When you write <Person> alice, bob, two things happen:

  1. Person is declared as an element (in <element>).
  2. alice and bob are declared as elements that are in Person.

Containers are elements too. You can nest them, use them in facts, or make them members of other containers:

<element> Animal
<Animal> Cat, Dog
<Cat> whiskers, mittens

# Cat is in Animal, whiskers is in Cat
# All of them are in element

Membership Constrains Relations

When a relation declares a signature like rel R/2 : Person, City, only elements that are in Person can appear in argument 1, and only elements in City can appear in argument 2. This filtering applies to user-written facts AND to derived facts.

Reflexivity respects membership. If rel eq/2 : Person, Person, reflexive, then eq(t, t) is only generated for elements in Person — not every element in the universe.

Relations

Named connections between terms.

A relation is a named predicate with a fixed arity (number of arguments). equiv/2 is a binary relation; between/3 is ternary.

Syntax

rel equiv/2                          # plain relation (untyped)
rel equiv/2 : equivalence              # with properties
rel lt/2 : reflexive, transitive       # multiple properties
rel lives_in/2 : Person, City         # membership constraints
rel friend/2 : Person, Person, symmetric # membership + properties

Properties

PropertyMeaningAuto-generated
reflexive R(t, t) for every constant t Facts: R(a, a), R(b, b), ...
symmetric R(x, y) implies R(y, x) Rule: R(?x, ?y) |- R(?y, ?x)
transitive R(x, y) and R(y, z) implies R(x, z) Rule: R(?x, ?y), R(?y, ?z) |- R(?x, ?z)
congruent If R(a, b) and S(...a...), then S(...b...) Substitution in existing facts
equivalence All four of the above Everything above combined
Equivalence is the strongest. Writing : equivalence is shorthand for : reflexive, symmetric, transitive, congruent.

See Elements for how membership restricts which elements each argument position accepts.

Facts

Ground atoms — the initial knowledge base.

A fact is a ground instance of a relation. All terms in a fact must be declared constants.

Syntax

equiv(a, b)
lt(x, y)
between(a, b, c)

Each fact is a single line: a relation name followed by parenthesized, comma-separated constants.

Validation

  • The relation must be declared with rel.
  • The number of terms must match the relation's arity.
  • Every term must be a declared constant.

If validation fails, the error message includes the line number.

← Relations Rules →

Rules

Inference rules for deriving new facts.

Rules express: "if these premises all hold, then derive these conclusions." They are the core mechanism for forward-chaining inference.

Syntax

rule name: premises |- conclusions

Premises and conclusions are comma-separated atoms. Variables are prefixed with ?.

Examples

# Symmetry: if R(x, y) then R(y, x)
rule sym: R(?x, ?y) |- R(?y, ?x)

# Transitivity: if R(x, y) and R(y, z) then R(x, z)
rule trans: R(?x, ?y), R(?y, ?z) |- R(?x, ?z)

# Cross-relation: friends know each other
rule f2k: friend(?x, ?y) |- knows(?x, ?y)

# Multi-conclusion: equivalence implies both directions
rule both: eq(?x, ?y) |- leq(?x, ?y), leq(?y, ?x)

How Variables Work

  • Variables (?x, ?y, etc.) are pattern wildcards that match any constant.
  • The same variable in multiple positions requires the same constant in all positions.
  • The engine finds all substitutions that satisfy all premises simultaneously, then instantiates conclusions.
  • Only ground conclusions (where every variable got bound) are added as new facts.
Multi-premise rules are expensive. A rule with N premises requires an N-way join over the fact set. Keep premise counts low for performance.

Derive

Compute the closure.

The derive command triggers the closure engine. It applies all rules repeatedly until no new facts can be derived (saturation) or a limit is reached.

Syntax

derive          # default: up to 100 rounds
derive 10       # limit to 10 rounds

What Happens

Each round, the engine:

  1. Applies all rules (auto-generated + user-defined) to the current fact set
  2. Generates reflexive facts for reflexive relations
  3. Applies congruence for congruent relations
  4. Checks if any new facts were added

Stops when:

  • Saturated — no new facts in a round (fixed point reached)
  • Round limit — maximum rounds exceeded
  • Fact limit — total facts exceed 10,000 (safety cap)
← Rules Algorithm →

Algorithm

How the closure engine works internally.

Forward Chaining

Relatum uses a semi-naive forward chaining strategy, similar to Datalog evaluation. Starting from initial facts, it repeatedly applies rules to derive new facts until a fixed point.

Round Structure

for each round:
  1. Rule application
     for each rule:
       find all substitutions matching premises
       instantiate conclusions with each substitution
       add new ground facts to the fact set

  2. Reflexivity
     for each reflexive relation R:
       for each constant t in universe:
         add R(t, t)

  3. Congruence
     for each congruent relation R:
       for each R(a, b) where a != b:
         for each existing fact S(...a...):
           add S(...b...) (and vice versa)

  4. Fixed-point check
     if no new facts: STOP (saturated)

Complexity

  • Single-premise rules: O(|rules| × |facts|) per round
  • Multi-premise rules: O(|facts||premises|) worst case (join)
  • Congruence: O(|pairs| × |facts|) per round
Beware combinatorial explosion. A transitive relation over many constants can generate O(n²) facts. The 10,000-fact safety cap prevents runaway computations.
← Derive Relation Properties →

Relation Properties

What each property actually does under the hood.

Reflexive

For every constant t in the universe, the engine adds R(t, t) before rule application begins. This includes constants that aren't mentioned in any user-supplied fact.

Symmetric

Auto-generates the rule R(?x, ?y) |- R(?y, ?x). Applied during normal rule matching each round.

Transitive

Auto-generates the rule R(?x, ?y), R(?y, ?z) |- R(?x, ?z). This is a two-premise rule, so it performs a join on the shared variable ?y.

Congruent

If R(a, b) holds and some fact S(..., a, ...) exists, the engine derives S(..., b, ...) (substituting a with b in each position, and vice versa). This generalizes equality to the entire fact base.

Equivalence = All Four

Declaring : equivalence is exactly the same as : reflexive, symmetric, transitive, congruent. It gives you a full equivalence relation that propagates through the entire system.

← Algorithm Safety & Limits →

Safety & Limits

Guarding against infinite derivation.

Built-in Limits

LimitDefaultPurpose
Max rounds100Prevents infinite rule application loops
Max facts10,000Caps memory usage from combinatorial explosion
Max substitutions per rule50,000Prevents single-rule blowup

When Limits Are Hit

The engine stops and reports the current state. The output panel shows a capped badge instead of saturated. The derived facts up to that point are still valid — they're just not complete.

Tips for Staying Safe

  • Start with few constants and add more gradually.
  • Use derive 5 to limit rounds while experimenting.
  • Be cautious with transitive + many constants — transitivity over n elements generates up to n² facts.
  • Congruence with many relations can also multiply quickly.
← Relation Properties Example: Equivalence →

Example: Equivalence Closure

The canonical Relatum example.

<element> a, b, c

rel equiv/2 : equivalence

equiv(a, b)
equiv(b, c)

derive

What the engine derives

Starting from equiv(a, b) and equiv(b, c):

  • Symmetry: equiv(b, a), equiv(c, b)
  • Transitivity: equiv(a, c) (via a-b, b-c)
  • Symmetry of derived: equiv(c, a)
  • Reflexivity: equiv(a, a), equiv(b, b), equiv(c, c)

The engine saturates in 2-3 rounds with 9 total facts (the complete 3×3 equivalence class).

← Safety & Limits Example: Partial Order →

Example: Partial Order

Reflexive + transitive, but NOT symmetric.

<element> a, b, c, d

rel leq/2 : reflexive, transitive

leq(a, b)
leq(b, c)
leq(c, d)

derive

What the engine derives

  • Reflexivity: leq(a, a), leq(b, b), leq(c, c), leq(d, d)
  • Transitivity: leq(a, c), leq(a, d), leq(b, d)

Note: leq(b, a) is not derived because the relation is not symmetric. This is exactly what you want for a partial order.

← Equivalence Example: Custom Rules →

Example: Custom Rules

User-defined inference beyond built-in properties.

<element> alice, bob, carol

rel friend/2 : symmetric
rel knows/2

# If X is friends with Y, then X knows Y
rule f2k: friend(?x, ?y) |- knows(?x, ?y)

# Knowing is transitive
rule knows_trans: knows(?x, ?y), knows(?y, ?z) |- knows(?x, ?z)

friend(alice, bob)
friend(bob, carol)

derive

Derivation chain

  1. Symmetry: friend(bob, alice), friend(carol, bob)
  2. f2k rule: knows(alice, bob), knows(bob, alice), knows(bob, carol), knows(carol, bob)
  3. knows_trans: knows(alice, carol), knows(carol, alice), knows(alice, alice), etc.

The result: everyone knows everyone — because friendship is symmetric and "knowing" is transitive, the entire network collapses into full connectivity.

← Partial Order Example: Congruence →

Example: Congruence

How congruence propagates equivalence through other relations.

<element> a, b, c

rel equiv/2 : equivalence
rel R/2

equiv(a, b)
R(a, c)

derive

What congruence does

Because equiv is congruent and equiv(a, b) holds:

  • The engine sees R(a, c) and substitutes a → b, deriving R(b, c).
  • If R(c, a) existed, it would also derive R(c, b).

Congruence is the key mechanism that makes equivalence relations useful — it propagates the "sameness" of a and b through the entire fact base.

Example: Membership

Restricting which elements each relation argument accepts.

<Person> alice, bob, carol
<City> paris, london

# lives_in: arg 1 must be in Person, arg 2 must be in City
rel lives_in/2 : Person, City

# friend: both args must be in Person, and it's symmetric
rel friend/2 : Person, Person, symmetric

# same_city: equivalence, but only on City
rel same_city/2 : City, City, equivalence

lives_in(alice, paris)
lives_in(bob, paris)
lives_in(carol, london)
friend(alice, bob)

rule shared: lives_in(?x, ?c), lives_in(?y, ?c) |- same_city(?c, ?c)

derive

What happens

  • Membership prevents nonsense: lives_in(paris, alice) would be rejected because paris is not in Person.
  • Reflexivity respects membership: same_city(t, t) is only generated for paris and london — not for alice, bob, or carol.
  • Rules are membership-filtered: if a rule would derive a conclusion that violates membership constraints, it is silently dropped.
  • Container names are elements: Person and City are in the universe. You can write has_type(alice, Person) with an unconstrained relation.
Membership is lightweight. It is purely a filtering mechanism — it doesn't change the engine's algorithm, it just restricts which facts can exist.

Rust API

Using Relatum as a library in your Rust project.

Relational Engine

use relatum::relational::*;

// Create engine and declare entities
let mut engine = ClosureEngine::new();
let a = engine.define_constant("a");
let b = engine.define_constant("b");
let c = engine.define_constant("c");

// Define equiv as a full equivalence relation
engine.define_equivalence("equiv");

// Add ground facts
engine.add_fact(Relation::binary("equiv", a.clone(), b.clone()));
engine.add_fact(Relation::binary("equiv", b, c.clone()));

// Derive closure
let result = engine.derive_closure();

assert!(result.facts.contains(
    &Relation::binary("equiv", a.clone(), c.clone())
));
assert!(result.saturated);

Entity & Relation Declarations

// Constants are ground terms in the universe
let x = engine.define_constant("x");

// Variables are pattern wildcards for rules
let v = engine.define_variable("v");

// Declare a relation schema with arity
engine.define_relation("lt", 2);

// Mark properties individually
engine.mark_reflexive("lt");
engine.mark_congruent("lt");

// Or use define_equivalence for the full package
engine.define_equivalence("eq");

Custom Rules

use relatum::relational::rule::*;

// Build a custom rule: friend(x, y) |- knows(x, y)
let rule = Rule::new(
    "friend_implies_knows",
    vec![RelationPattern::new("friend", vec![
        Term::var("x"), Term::var("y")
    ])],
    vec![RelationPattern::new("knows", vec![
        Term::var("x"), Term::var("y")
    ])],
);
engine.add_rule(rule);

Closure Result

let result = engine.derive_closure();

// All facts (initial + derived)
for fact in &result.facts {
    println!("{}", fact);
}

// Only newly derived facts
println!("Derived {} new facts", result.derived.len());
println!("Saturated: {}", result.saturated);
println!("Rounds: {}", result.rounds);
